Accessibility in the Development Process

What is Accessibility

  • Access to web products for users who may utilize assistive technologies
    • Screen readers
    • Braille computers
    • Keyboard (non-mouse)
    • Voice Control (dragon speaking naturally)
    • Many more!
  • Common considerations
    • Blindness
    • Hearing loss
    • Mobility loss
    • Colorblindness
    • Limited vision
    • Dexterity issues in fingers/arms
    • Cognitive imparements
    • Many more!

The point is not to categorize, but to hold our products to a higher standard for inclusivity.

Web Accessibility

Refers to standards set by Web Content Accessibility Guidelines (WCAG)

Include guidelines for content, design, coding, and testing best practices

“How would development know?”

Developers! Own your process!

Web accessibility is tied directly to the quality of your product.

Champion and take responsibility for accessibility specs.

  • Document your process (give yourself credit)
  • Deliver results that are testable and trackable (lasting impression)
  • Gain experience (open conversations with other roles further up the pipeline)

Quick code reviews

  • Tooltips, accordions, anything changing state with JavaScript
    • How will that change be announced?
      • aria-live, aria-expanded, role=alert
    • How can users open/close without clicking?
  • Elements only distinguished by color
    • Links in paragraphs of text
    • Status changes (complete, error, warning etc)

How can we avoid the 80 page post launch audit?

  • Code Reviews
  • Automated Tests
  • Pattern libraries (Brad Frost's patternlab)

How to push back.

  • Hierarchy of the DOM vs UI?
  • Are the designs or wireframe missing accessibility requirements? (Search Example
